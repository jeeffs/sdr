🔴 Bugs Críticos — Correção Imediata
CRÍTICO
C1 — Hash do Admin em localStorage SEM SALT — Rainbow Table Attack
Confirmado em produção via inspeção de localStorage: sdr_admin_bio_fhash_master = edbf7822b714ae27955bc2059896bdeed7eb290c6c2cbb9c7d0358a979f4f02c — SHA-256 puro sem salt, armazenado permanentemente. Qualquer XSS ou acesso físico ao computador expõe o hash do admin. Rainbow tables on-line crackam SHA-256 simples em segundos.
// localStorage — encontrado em produção: "sdr_admin_bio_fhash_master": "edbf7822...f02c" // SHA-256 sem salt — PERIGO // O mesmo pattern no mobile: "sdr_bio_fhash": uid + ':' + h // hash em sessionStorage durante toda a sessão
Solução: (1) Nunca salvar hash de senha em localStorage/sessionStorage. (2) Substituir por Firebase Auth token (curto prazo de validade). (3) Para biometria: salvar apenas o credentialId WebAuthn, nunca o hash. (4) Migrar imediatamente: localStorage.removeItem('sdr_admin_bio_fhash_master') + recriar fluxo com Firebase token.
CRÍTICO
C2 — Git Repository CORROMPIDO — Deploy em Risco
O diretório .git/ contém dezenas de arquivos index.lock.* com nomes extremamente longos (ex: index.lock.gone_c.old.purge.z.gz.del.x.x2.x2...). Isso indica que múltiplas tentativas de operação git falharam e deixaram locks. O repositório está em estado inconsistente. Qualquer git commit pode falhar silenciosamente ou corromper o histórico.
// Arquivos encontrados em .git/: index.lock.1.gone_c.old.purge.z.gz.del.x.x2.x2.x2.x2... index.lock.2.gone_c.old.purge.z.gz... index.lock.bak2.gone.gone_c... // (dezenas de variações) // ORIG_HEAD e MERGE_MSG.swp presentes — merge incompleto!
Solução: (1) rm -f .git/index.lock* para limpar todos os locks. (2) git fsck --full para verificar integridade. (3) git status e resolver qualquer conflito pendente. (4) Fazer backup completo ANTES de qualquer operação. (5) Configurar .gitignore para excluir arquivos temporários do Windows.
CRÍTICO
C3 — firebase:previous_websocket_failure — Firebase WebSocket Falhou em Produção
Confirmado em localStorage: firebase:previous_websocket_failure: "true". O SDK Firebase tentou conexão WebSocket e falhou, forçando o monkey-patch REST. Isso cria dois problemas: (1) listeners .on() do SDK não recebem dados em tempo real; (2) cada leitura vira um HTTP GET independente = mais cobranças no RTDB + latência maior. Esta pode ser a causa da cobrança alta reportada ontem.
// localStorage em produção: "firebase:previous_websocket_failure": "true" // Impacto no código: // db.ref('users').once('value') → HTTP GET /users.json?auth=TOKEN // db.ref('os').once('value') → HTTP GET /os.json?auth=TOKEN (11.456 registros!) // Sem cache → cada reload do admin relê TUDO do banco
Solução imediata: (1) Verificar no Firebase Console → Realtime Database → Usage se as leituras explodiram. (2) Implementar cache client-side com TTL de 5min para dados que não mudam frequentemente (precos, config, users). (3) Paginar a leitura de OS — não carregar todos os 11.456 de uma vez. (4) Investigar por que o WebSocket está falhando (firewall corporativo? CORS?)
CRÍTICO
C4 — Duplo Bônus no Financeiro Mobile do Supervisor (Juliano)
Bug documentado no CLAUDE.md e SPRINT_LOG_MAIO_2026.md mas NÃO CORRIGIDO. Na função renderFinanceiro do index.html, o bloco do bônus usa condição isV2 em vez de isV2 && !isSupervisor. Resultado: o card "Coordenação Técnica" (R$88,25) aparece duplicado no Resumo Financeiro do Juliano.
// BUG em index.html — renderFinanceiro: if (isV2) { // ← ERRADO: mostra bônus para supervisor mesmo que já incluso html += `<div class="card purple">Coordenação Técnica: ${fmt(bonus)}</div>`; } // CORRETO: if (isV2 && !isSupervisor) { // bônus extra só para V2 não-supervisor html += `...`; }
Solução: Em public/index.html, na função renderFinanceiro, alterar a condição do bloco do bônus de if (isV2) para if (isV2 && !isSupervisor). Bumpar SW cache e fazer deploy.
CRÍTICO
C5 — allRecords no Mobile Carrega Só Dados do Usuário Logado
O supervisor (Juliano) precisa de OS de TODA a equipe para calcular totalGestao (diferencial V2-V1). Mas allRecords no mobile filtra apenas os registros do usuário logado. O total exibido no card "Coordenação Técnica" está incorreto — mostra apenas as próprias OS do Juliano multiplicadas pelo diferencial, não o diferencial de toda equipe.
// index.html — loginSuccess: allRecords = records.filter(r => r.userId === currentUser.id); // ↑ Filtra só o próprio Juliano — supervisor precisa de TODOS // Correto para supervisor: allRecords = isSupervisor ? records // todos os técnicos : records.filter(r => r.userId === currentUser.id);
Solução: Após determinar isSupervisor no loginSuccess, carregar todos os registros se o técnico for supervisor, filtrado apenas pelo mês corrente para não explodir memória/leitura.
CRÍTICO
C6 — loginComo Sem Verificação de Sessão Master Ativa (Mobile)
O parâmetro ?loginComo=uid&by=masterUid no index.html (linha ~1253) faz impersonate imediato se o by existir no usersCache com role === 'master'. Porém: (1) usersCache é carregado antes do login — qualquer um pode criar uma URL com um uid master existente. (2) Não há verificação de sessão Firebase ativa do master. O fix foi parcial.
Solução: Exigir que o parâmetro ?loginComo venha acompanhado de um token Firebase válido do admin (gerado server-side via Cloud Function), não apenas um UID no banco.
CRÍTICO
C7 — Token Firebase Exposto na URL (REST API)
O monkey-patch REST em index.html adiciona o token JWT como query string: /os.json?auth=TOKEN. Tokens JWT em URLs são logados em: servidores de proxy corporativos, nginx/apache logs, CDN logs, histórico do browser, Referer headers de terceiros. Qualquer técnico com acesso ao servidor ou proxy pode capturar tokens válidos.
// index.html — REST patch: const url = `${REST_BASE}/${path}.json?auth=${tok}`; // ↑ TOKEN VISÍVEL NA URL // Correto — token no header: fetch(url, { headers: { 'Authorization': `Bearer ${tok}` } }) // Firebase RTDB REST aceita Authorization: Bearer
Solução: Alterar todas as chamadas REST no monkey-patch (index.html e sdr-bundle.js) para mover ?auth=TOKEN para o header Authorization: Bearer TOKEN. Firebase RTDB REST suporta ambos os formatos.
CRÍTICO
C8 — usersCache Carrega Hashes de Todos os Técnicos Antes do Login
O mobile carrega db.ref('users').once('value') ANTES da autenticação, trazendo passwordHash, salt, dados bancários (conta, agência, PIX, CPF, RG) de TODOS os 20 técnicos. Qualquer código JS na página pode ler esses dados. Se houver XSS, atacante obtém credenciais de todos.
Solução: Antes do login, carregar apenas name e passwordHash + salt do usuário específico (após ele digitar o nome). Dados completos de outros técnicos só devem ser carregados pelo admin após autenticação.
🟠 Problemas Altos — Próximo Sprint
ALTO
A1 — Inventário e Ficha EPI Não Renderizam Conteúdo
Testado em produção: pg-inventario renderiza apenas 310 caracteres (página quase vazia) e pg-ficha-epi apenas 332 caracteres. O hook showPage não está chamando as funções de render dessas páginas quando o usuário navega até elas. O técnico vê uma página em branco.
Solução: Verificar no patch de showPage (index.html) se os casos 'pg-inventario' e 'pg-ficha-epi' estão presentes. Se não, adicionar if (name === 'pg-inventario') renderInventario(); e similar para EPI.
ALTO
A2 — corrigirPrecosDoMes Grava Total Errado na OS
Após executar "Corrigir Preços do Mês" no admin, o campo total da OS é calculado somando apenas sv.total dos serviços corrigidos, ignorando os demais. Todos os cálculos financeiros subsequentes ficam errados para OSs corrigidas.
// BUG — sdr-bundle.js: const novoTotal = servicosCorrigidos.reduce((s, sv) => s + (Number(sv.total)||0), 0); // sv.total não existe para serviços não-corrigidos → contribuem com 0 // CORRETO: const novoTotal = servicosCorrigidos.reduce((s, sv) => s + (Number(sv.qtd)||0) * (Number(sv.valor)||0), 0);
Solução: Corrigir a redução para usar sv.qtd × sv.valor em vez de sv.total.
ALTO
A3 — Rate Limiting Apenas em Memória — Brute Force com F5
O rate limiting do admin (_loginAttempts) fica em memória. Um reload da página (F5) zera o contador, permitindo brute force ilimitado. O mobile tem rate limit persistido em localStorage (sdr_mob_rl encontrado em produção), mas o admin não tem essa proteção.
Solução: Persistir _loginAttempts em localStorage com TTL de 15 minutos, como já foi feito no mobile. Ou implementar Firebase App Check para bloquear requests não-browser.
ALTO
A4 — CSP com unsafe-inline + unsafe-eval — XSS Desprotegido
O firebase.json define CSP com 'unsafe-inline' e 'unsafe-eval' em script-src. Essas duas diretivas anulam completamente a proteção do CSP contra XSS. Qualquer script injetado por XSS executa normalmente.
// firebase.json — CSP atual (INSEGURO): "script-src 'self' 'unsafe-inline' 'unsafe-eval' https://..." // 'unsafe-inline' + 'unsafe-eval' = CSP inútil contra XSS // Alvo (requer migrar scripts inline para externos): "script-src 'self' 'nonce-{random}' https://www.gstatic.com https://cdnjs.cloudflare.com"
Solução longa: Mover todos os scripts inline para arquivos externos (sdr-bundle.js já é o caminho certo). Remover 'unsafe-eval' verificando quais libs CDN realmente precisam (geralmente nenhuma). Usar nonce ou hash no CSP.
ALTO
A5 — Sem Fila Offline de OS — Perda de Dados em Campo
Se um técnico lançar uma OS sem rede, o salvarOS() falha e a OS é perdida sem retry. O patch REST substituiu o SDK Firebase, que tinha keepSynced nativo para offline. O técnico nem vê mensagem de erro clara — simplesmente perde o trabalho.
Solução: Implementar fila em IndexedDB: gravar localmente primeiro com status "pendente", exibir badge "1 OS aguardando sync", sincronizar via listener window.addEventListener('online', ...).
ALTO
A6 — Dados Bancários e CPF/RG em Texto Puro no Firebase
Campos emp-banco, emp-agencia, emp-conta, emp-pix, emp-resp-cpf, emp-resp-rg são gravados em plaintext em users/{uid}. Esses dados são sensíveis pela LGPD (art. 5º, II) e qualquer vazamento do RTDB os expõe diretamente. CPF é dado sensível — exige atenção especial.
Solução: (1) Criptografar via Web Crypto API (AES-GCM) antes de gravar. (2) Ou mover para subcoleção separada com regras RTDB mais restritivas. (3) Adicionar aviso de privacidade (LGPD art. 8º) no formulário de dados empresariais.
ALTO
A7 — Service Worker: Versão no Comentário Desatualizada
O sw.js tem comentário // v132 no topo, mas CACHE_NAME = 'sdr-v148'. Além disso, o log de install está hardcoded como '[SDR SW] install — v42' em vez de referenciar CACHE_NAME. Dificulta diagnóstico de problemas de cache em produção — o console mostra uma versão que não bate com nada.
Solução: Atualizar comentário do arquivo para // sdr-v148. Mudar console.log('[SDR SW] install — v42') para console.log('[SDR SW] install —', CACHE_NAME).
ALTO
A8 — Exceção Jefferson Implementada de 4 Formas Diferentes
Cada módulo identifica Jefferson de forma diferente: importar.js usa .includes('JEFFERSON') (captura "Carlos Jefferson" errado), admin.html usa === 'jefferson', dashboard_admin.js usa === 'jefferson', financeiro.js usa .includes('jefferson'). Comportamento inconsistente entre módulos.
Solução: Centralizar em campo excecaoV3: true no Firebase users/jefferson/excecaoV3. Todos os módulos consultam esse flag em vez de comparar nomes hardcoded.
ALTO
A9 — Sem SRI nas Bibliotecas CDN — Supply Chain Attack
Leaflet, Font Awesome, Tesseract.js, qrcode-generator e ExcelJS são carregados de CDN sem atributo integrity (SRI — Subresource Integrity). Se o CDN for comprometido, JS malicioso é entregue a todos os técnicos sem que o browser detecte. Risco real: cdnjs.cloudflare.com já teve incidentes.
Solução: Adicionar integrity="sha384-..." e crossorigin="anonymous" em cada <script> e <link> CDN. Gerar os hashes via openssl dgst -sha384 -binary FILE | openssl base64 -A.
ALTO
A10 — Tesseract.js (~2.5MB) Carregado para Todos no Head
A biblioteca de OCR Tesseract.js (~2.5MB) é carregada sincronicamente no <head> de index.html para todos os técnicos, mesmo os que nunca usam OCR. Isso aumenta o tempo de carregamento inicial em ~1-2 segundos em redes 3G (campo). Piora significativa da experiência do técnico em campo.
Solução: Lazy-load com import() dinâmico apenas quando o recurso de OCR for acionado. Se o recurso não estiver sendo usado atualmente, remover completamente.
ALTO
A11 — Leitura de Todos os 11.456 OS a Cada Login no Admin — Cobrança Alta
O admin faz db.ref('os').once('value') carregando todos os registros a cada login. Com 11.456 OS e crescendo, isso gera cobranças significativas no Firebase RTDB (plano Blaze cobra por GB baixado). Este provavelmente é o motivo da "cobrança alta" reportada ontem — além do firebase:previous_websocket_failure que faz o SDK tentar WebSocket E depois REST.
// Impacto estimado: // 11.456 OS × ~500 bytes/OS = ~5.7 MB por leitura completa // Se 10 logins/dia no admin = ~57 MB/dia de leituras // Firebase cobra $1/GB — parece barato mas escala // firebase:previous_websocket_failure = true // → SDK tenta WebSocket (falha) → depois REST (cobra) // → 2x tentativas por operação
Solução: (1) Paginar leitura de OS: carregar apenas o mês atual por padrão. (2) Implementar cache em memória com TTL de 5min. (3) Investigar e corrigir a falha do WebSocket (provavelmente firewall bloqueando WSS). (4) Usar limitToLast(200) na listagem inicial.
ALTO
A12 — Fingerprint de Duplicata Diverge Entre Preview e Execução
O fingerprint anti-duplicata usa campos diferentes no preview da importação vs. na execução real. Se preços mudarem entre duas importações do mesmo arquivo, o fingerprint não bate e a OS é importada como nova, gerando duplicatas no banco.
Solução: Normalizar fingerprint para usar apenas campos não-monetários: userId|tipo|cidade|cto|data|nSvs — excluindo o total que varia com reajuste de preços.

📊 Excel (.xlsx/.xlsm) vs CSV — Investigação dos Erros no Console

⚠️ Conclusão: Manter Excel para importação — CSV não é substituto aqui

As planilhas usadas pelos técnicos são arquivos .xlsm (com macros) que contêm fórmulas de cálculo. Converter para CSV perderia essas fórmulas. O sistema usa duas bibliotecas para leitura: XLSX (SheetJS) no browser e openpyxl (Python) no script de relatório.

ANÁLISE
Causa Real dos Erros no Console — Não é o Formato do Arquivo
Os erros de console reportados provavelmente são de: (1) WebSocket bloqueado — Firebase RTDB SDK tenta WSS e falha (confirmado pelo firebase:previous_websocket_failure). (2) CSP violations — scripts inline bloqueados em algumas situações. (3) Erro de SyntaxError em template literals (corrigido nos commits recentes). (4) Funções não definidas como MAX_SERVICOS_POR_OS (documentado e corrigido). O formato Excel em si não causa erros de console — a biblioteca SheetJS é estável.
// Erros típicos de console esperados no SDR: [Firebase SDK] WebSocket connection failed → REST fallback Content Security Policy: blocked inline script... ReferenceError: MAX_SERVICOS_POR_OS is not defined (corrigido) [SDR REST mobile] .once() erro: ... (rede instável em campo) // NÃO relacionados a Excel/CSV: // O SheetJS (XLSX) é confiável para .xlsx e .xlsm // openpyxl é confiável para Python scripts
Recomendação: (1) Manter .xlsx/.xlsm para importação de OS. (2) Usar CSV apenas para exportações simples (já implementado para Clientes e ONUs). (3) Investigar e corrigir o WebSocket Firebase (main cause dos erros). (4) Para o script Python, openpyxl está correto — não mudar.
OBSERVAÇÃO
O Script gerar_relatorio_sdr.py tem Caminhos Hardcoded que Podem Falhar
O arquivo gerar_relatorio_sdr.py referencia caminhos absolutos do Windows (C:\Users\Jeeff\OneDrive\...). Se rodado em outro computador ou se o OneDrive sincronizar em local diferente, todos os FILES falham com FileNotFoundError.
Solução: Usar pathlib.Path(__file__).parent / 'DADOS' / 'planilhas' para caminhos relativos ao script.
⚖️ Análise Jurídico-Trabalhista — Risco de Vínculo Empregatício

✅ O Que Está Correto nos Contratos

  • O contrato de prestação de serviços gerado pelo sistema (sdr-bundle.js) inclui Cláusula 4ª explícita: "não configurando, sob hipótese alguma, vínculo empregatício, subordinação, habitualidade ou pessoalidade" — adequado.
  • Referência correta ao art. 442 da CLT (prestação de serviços por PJ não gera vínculo) e arts. 593-609 do Código Civil.
  • Referência à Lei nº 14.063/2020 para validade da assinatura eletrônica.
  • Campo de empresa (CNPJ, Razão Social) existe no app — técnicos devem preencher como PJ.
  • O título dos documentos usa "Prestador Autônomo" e "Instrumento Particular de Parcelamento" — sem menção a salário, férias, FGTS, 13º.
  • O aditivo contratual menciona o prestador como "responsável pelo recolhimento de seus tributos, encargos sociais e previdenciários" — correto.

🚨 Pontos de Risco Trabalhista Detectados no Sistema

  • Habitualidade: 11.456 OS registradas indica prestação contínua e habitual — principal elemento caracterizador de vínculo (art. 3º CLT). Mitigar com intervalos, múltiplos clientes.
  • Subordinação via Supervisor: O técnico Juliano "coordena" e "supervisiona" a equipe, recebendo diferencial financeiro. Essa estrutura hierárquica dentro do sistema pode ser interpretada como subordinação direta. Evitar linguagem de "supervisor" — preferir "coordenador de projetos autônomo".
  • Controle de Jornada: O campo "Hora" nas OS registra horário preciso de cada serviço. Se os horários forem consistentemente dentro de uma janela fixa (08h-17h), isso pode configurar controle de jornada — elemento de vínculo. O sistema deveria apenas registrar a OS, não controlar horário.
  • Pessoalidade: UIDs individuais (tecnico1, tecnico2...) vinculam serviços à pessoa física, não à empresa. Idealmente, o login deveria ser pelo CNPJ/empresa, não pelo nome pessoal.
  • Exclusividade Econômica: O supervisor recebe percentual de toda a equipe — criando dependência econômica do sistema SDR. Se o técnico prestar serviço 100% para uma empresa, Justiça do Trabalho presume relação de emprego (Súmula 331 TST).
RISCO ALTO
J1 — Campo "Hora" nas OS Pode Configurar Controle de Jornada
O formulário "Nova Ordem de Serviço" tem campo obrigatório "Hora" (input type="time"), preenchido automaticamente com a hora atual. Registros de OS mostrando técnicos sempre trabalhando das 08h às 17h podem ser usados em reclamação trabalhista como prova de controle de jornada.
Recomendação jurídica: (1) Tornar o campo "Hora" opcional. (2) Adicionar disclaimer: "Hora de execução do serviço (autonomamente definida pelo prestador)". (3) Ou remover o campo de hora — registrar apenas a data da OS.
RISCO ALTO
J2 — Termo "Supervisor" Cria Aparência de Subordinação Hierárquica
O sistema usa "Supervisão Técnica", "Supervisor", "coordena a equipe" para o Juliano. A CLT define subordinação como o poder do empregador de dirigir, controlar e disciplinar a prestação de serviços. Uma auditoria trabalhista que veja "Supervisão: true" no banco pode argumentar subordinação.
Recomendação jurídica: (1) Renomear no banco e no UI: "supervisao" → "coordenador_projeto". (2) Contrato do Juliano deve especificar que a coordenação é responsabilidade PJ dele, não poder de direção da contratante. (3) O diferencial V2-V1 deve ser chamado de "comissão por resultado da equipe" ou "bônus de coordenação por projeto".
RISCO MÉDIO
J3 — UIDs Pessoais Vinculam Serviços à Pessoa Física, Não à Empresa
Os UIDs tecnico1, tecnico2, jefferson são nomes pessoais, não empresariais. Idealmente, o contratante paga o CNPJ/empresa, e a empresa (MEI/ME) presta o serviço. O sistema deve exibir e registrar pelo nome da empresa quando disponível.
Recomendação: Exibir no admin e nos relatórios a Razão Social do prestador (já coletada no campo emp-razao-social), não o nome pessoal. Garantir que todos os técnicos tenham o campo empresa preenchido.
RISCO MÉDIO
J4 — Ausência de Cláusula de Não-Exclusividade Explícita
O contrato gerado não tem cláusula de não-exclusividade. A Súmula 331 do TST considera que exclusividade é elemento de vínculo. Documentar que o prestador pode e deve atender outros clientes é proteção adicional.
Recomendação: Adicionar ao template do contrato: "Cláusula X — Da Não-Exclusividade: O PRESTADOR é livre para prestar serviços a outras pessoas físicas ou jurídicas, não havendo qualquer restrição de exclusividade neste instrumento."
RISCO MÉDIO
J5 — LGPD: CPF e RG Coletados Sem Aviso de Privacidade
O formulário de dados empresariais coleta CPF e RG (dados sensíveis conforme LGPD art. 5º, II) sem exibir aviso de privacidade, base legal ou finalidade do tratamento. A ANPD pode autuar com multas de até 2% do faturamento (máx. R$50M).
Recomendação: Adicionar antes do formulário: "Seus dados são coletados para fins contratuais (art. 7º, V, LGPD) e serão tratados com confidencialidade. Não serão compartilhados com terceiros." + link para Política de Privacidade.

💰 Análise da Cobrança Alta no Firebase RTDB — Mudança de Ontem
URGENTE
Causa Identificada: WebSocket Failure + Leituras Sem Cache
Dois fatores combinados explicam a cobrança alta:

1. firebase:previous_websocket_failure = true (confirmado em localStorage de produção): O SDK Firebase tenta conectar via WebSocket (wss://*.firebaseio.com) e falha. Isso gera uma tentativa de conexão TCP + handshake + timeout para CADA operação, antes do fallback para REST. Multiplica o número de requests.
